Designers need a prototype; developers need a styleguide.

A prototype is not a styleguide. A prototype is a lorum ipsum sample site, which is great for other developers and stakeholders to get a feel for the site, but not all we as developers need. We need a styleguide to isolate each component and narrow down our requirements. A styleguide also helps group similar components together so we can strip down the similar but not identical elements. A styleguide is like a kitchen sink. It looks nothing like a sample page but it has everything the sample page needs.

Stylesheet order really does matter.

While it may sometimes seem logical to write your CSS to match the DOM, it actually hinders long-term development. Your stylesheet should be an inverted triangle, starting with very generic and far-reaching selectors, moving to very explicit, localized selectors, and finally ending with trumps and hacks. This makes development much more maintainable in the long run for many reasons:

  1. Specificity is kept as low as possible and is much easier to override later on. Throughout the stylesheet, specificity increases, never (hardly ever) decreases.
  2. You are grouping similar components together, not components that just so happen to live next to each other in the DOM.
  3. Any future additions to the CSS has a predictable place, and future removals are easier to find.

Why does this matter? Reusability. You can reuse a .main-link anytime you want. You can’t necessarily put a header > nav#menu ul li a anywhere on the page.

Documentation should be directly in the CSS as comments.

Unfortunately there is no equivalent to JavaDoc for CSS (although some proposals have been made). That doesn’t mean you and your team can’t agree on some commenting convention. Generally, there should be 3 levels of comments:

  1. architecturally: table of contents and headers for sections of CSS source code
  2. component-wise: each object, component, or thing on a page has a class that deserves its own comment, explaining when, how, and why it should be used.
  3. developer-level: small, one-line comments explaining why something was done. Always do this when you use !important, but you can also do this for code that isn’t self-explanatory. For example, what does this 18px come from?
