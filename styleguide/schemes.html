Site-Wide Schemes

Typeface Scheme

Avenir Next

Used for the main body copy, Avenir Next is a sans-serif type with little to no thick/thin transition. Being a sans-serif, its italic is actually an oblique, which just slants the glyphs.
Similar: Helvetica Neue
.a-ff-base

Optima

Optima is used only for headings of body copy (not table headings, etc.). It is a sans-serif polyweight type with a moderate thick/thin transition. Because of this, it goes well with both monoweight types and serif types, which makes it perfect for headings.
Similar: Seravek, Tahoma.
.a-ff-heading

Baskerville

Baskerville is used for text in formal documents and articles. It is a serif oldstyle type, which is easy on the eyes and can be read at extensive length. Like the heading font family, it, too, has a moderate thick/thin transition, but has diagonal stress and serifs, and true italics, which means it uses a whole new set of glyphs.
Similar: Palatino, Cochin1
.a-ff-prose

  • 1 Baskerville, though considered an oldstyle type, actually has a straight, vertical stress, which is typically a feature of modern types. However Palatino and Cochin indeed have diagonal stresses.

Helvetica Neue

Helvetica Neue is used for typographical and UI components such as ledes, pull quotes, captions, alerts, labels, buttons, and keys. Like Avenir, it is a monoweight sans-serif, yet slightly heavier.
Similar: Helvetica
.a-ff-accent

Menlo

Menlo is a monospace type used for sample code on the site. Personally, I dislike monospace fonts with serifs, so I chose a sans-serif.
Similar: Monaco, Andale Mono
.a-ff-code

Didot

Didot is a serif modern type use for mathematical spans and spans in foreign languages. It has a radical thick/thin transition, vertical stress, and straight horizontal serifs. With a cold yet elegant look, this font family makes long spans of text hard to read, but that’s okay because this font is used only for short expressions.
Similar: Bodoni 72, Times
.a-ff-math

Zapfino

Zapfino is used for very fancy elements such as drop-caps and subheadings. Use this font family sparingly.
.a-ff-script

Font Sizes

The first list shows atoms prefixed fz-, which change only font size, mainly used for inline (text-level) elements. These atoms use em units, so the font sizes will scale according to their parents. The second lists atoms prefixed fzb-, for font-size/line-height combos, which are useful for maintaining vertical rhythm on block elements. These use rem units, which are absolute. Only the first list is demonsrated (the second will look exactly the same save the line height).

Notice there are two pairs of classes for each size. The first classname has an SI prefix that corresponds to a power of 1000, while the second classname uses a Greek/Latin prefix that indicates the value of font-size in its definition. The first set is good for comparing two different atoms, and the second set is good for when you need to know the absolute size.

  • .a-fzb-tera , .a-fzb-tet (4.00/0.375)
  • .a-fzb-giga , .a-fzb-tri (3.00/0.50)
  • .a-fzb-mega , .a-fzb-di (2.00/0.75)
  • .a-fzb-kilo , .a-fzb-sesqui (1.50/1.00)
  • .a-fzb-normal (1.00/1.50)

Overlapping Opacity

Calculating the Perceived Alpha Value

The common intuition that overlapping alphas are simply added is incorrect. For example, a block with an opacity of 0.4 in front of another block with an opacity of 0.6 does not result in a total opacity of 1.0. It actually results in an opacity of 0.76. The fact of the matter is that no matter how many translucent blocks overlap (as long as their alphas are less than 1), you will never get something completely opaque. (You may get very close, like 0.99991, but not exactly 1.0.) This implies that the model for overlapping opacities is some sort of exponential decay1 that increasingly and asymptotically approaches 1.0.

To cut to the chase, the formula used to calculate the final resulting alpha, call it \alpha, from a given number of alphas, \alpha_1, \alpha_2, \dots, \alpha_n, is

\alpha = 1 - \bigl[\left(1-\alpha_1\right)\left(1-\alpha_2\right)\cdots\left(1-\alpha_n\right)\bigr]

Therefore the true resulting opacity from two overlapping blocks with opacity 0.4 and 0.6 is

\begin{align} \alpha &= 1 - \left(1-0.6\right)\left(1-0.4\right) \\ &= 1-\left(0.4\right)\left(0.6\right) \\ &= 1-0.24 \\ &= 0.76 \end{align}

If all of the alphas are a constant \alpha_1, however, we may convert this formula into a function of one variable: the number of overlapping blocks. This results in our desired increasing exponential decay function. Indeed, its right-hand limit is 1.

\alpha(n;\alpha_1) = 1 - \left(1-\alpha_1\right)^n

  • 1 I call it exponential decay because the base of the exponent is less than 1. This is true even if the function itself is increasing. It would only be called exponential growth if the base were more than 1, again, whether the function were increasing or decreasing.
